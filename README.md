## Overview

The **SIGC DD API** is a high-performance RESTful backend designed for the complete commercial management cycle of Dulces y Dulces S.A. It centralizes sales operations, sales-force quota tracking, client and product masters, and provides a bulk-import engine capable of ingesting large-volume transactional files (TSV) generated by the ERP.

Key capabilities:

- Full CRUD management of 25+ commercial entities.
- Hierarchical product catalogue (Mega-category → Category → Sub-category → Item).
- Geographic structure (City → Neighborhood) linked to clients and sales channels.
- Sales-force quota engine at daily, weekly, and monthly granularity.
- Batch import of **millions of sales records** in a single operation with real-time progress tracking.
- bcrypt-secured user authentication with role-based access control.

## Functional Modules

### Security & Authentication

**Base route:** `/api/auth`

Authentication is credential-based (vendor code + password). Passwords are stored as **bcrypt** hashes (10 salt rounds). Plain-text passwords are transparently migrated to bcrypt on first successful login.

| Endpoint | Method | Description |
|---|---|---|
| `/api/auth/login` | `POST` | Authenticate with `codigo`/`username` + `password`. Returns vendor profile and role. |
| `/api/auth/register` | `POST` | Register a new user account linked to an existing vendor record. |
| `/api/auth/register/bulk` | `POST` | Batch-register multiple users in a single request. |

**User model fields:** `id_usuario`, `username`, `password` (bcrypt), `estado` (active/inactive), `id_rol` (FK → `roles`).

**Roles model:** Flexible role system. Default role resolves to `Vendedor` if no `id_rol` is supplied.

## Optimized Import System

**Route:** `POST /import/ventas`  
**Status:** `GET /import/status`

The import system is designed for high-throughput ingestion of TSV sales files exported from the corporate ERP.

### How It Works

1. A TSV file is uploaded via multipart form to `POST /import/ventas`.
2. The server streams the file line-by-line using Node.js `readline` (no full in-memory load).
3. **Pre-loading phase:** All master tables (providers, categories, clients, items, channels, cities, etc.) are loaded into in-memory `Map` structures for O(1) lookup during processing.
4. **Upsert logic:** Each data row resolves master foreign keys. If a referenced master record does not exist, it is created automatically.
5. **Batch insertion:** Sales headers and line items are collected into batches and inserted using `bulkCreate` for maximum throughput.
6. **Transaction management:** Rows are committed in configurable transaction windows to balance durability and performance.

### Standard vs. Optimized Import

| Feature | Standard (`importventas.js`) | Optimized (`importventas-optimizado.js`) |
|---|---|---|
| Master pre-loading | Per-row DB lookups | Single pre-load into Maps at start |
| Batch size | Row-by-row | **10,000 rows / batch** |
| Transaction window | Per-row | **5,000 rows / transaction** |
| Progress reporting | Every row | Every **5,000 rows** |
| Suitable for | Small test files | Production / large files |

### TSV Column Mapping

The importer expects the following 38 columns (tab-separated):

```
LINEA · CATEGORIA · CANAL · Codigo vendedor · Nombre vendedor · Nro documento
Item · Desc. item · U.M. Orden · Fecha · Cliente factura · Sucursal factura
Razon social cliente factura · Direccion 1 · Desc. ciudad · Cantidad emp.
Cantidad · Costo promedio total · Valor bruto · Valor descuentos · Valor subtotal
Valor impuestos · Valor neto · Margen promedio · Impuesto afecta margen
Factor U.M. emp. · Factor U.M. Orden · Peso en KILO · Barrio
DETALLE TIPO DE NEGOCIO · Cond. pago fact · Nombre establecimiento  facturar
TIPO DE NEGOCIO · SUBCANAL · SUBCANAL DETALLADO · MEGACATEGORIA · SUBCATEGORIA
REPORTE PROV CON OBS
```

> **Date format expected:** `DD/MM/YYYY` (local date, no timezone conversion — dates are stored as-is in the `DATEONLY` Sequelize type)  
> **Currency format expected:** `$X.XXX,XX` — Colombian Peso (COP) using period as thousands separator and comma as decimal separator (e.g., `$1.250,50`)

## Developer Guide

### Prerequisites

- **Node.js** ≥ 18.x
- **PostgreSQL** ≥ 16 (or a compatible service such as NeonDB)
- **npm** ≥ 9.x

### Installation

```bash
# 1. Clone the repository
git clone https://github.com/AsllyZuniga/API_Sistema_Integral_de_Gestion_Comercial_Dulces_y_Dulces_SIGC_DD.git
cd API_Sistema_Integral_de_Gestion_Comercial_Dulces_y_Dulces_SIGC_DD

# 2. Install dependencies
npm install

# 3. Configure the database connection
cp config/config.example.json config/config.json
# Edit config/config.json with your database credentials

# 4. Set environment variables (create a .env file in the project root)
cp .env.example .env   # if .env.example is present, otherwise create manually
# Edit .env as described below

# 5. Run database migrations
npx sequelize-cli db:migrate

# 6. Start the server
npm start
```

The server listens on `http://localhost:3000` by default.

### Environment Variables

Create a `.env` file in the project root with the following variables:

```dotenv
# Server
PORT=3000
NODE_ENV=development

# Database (used by Sequelize if not relying on config/config.json directly)
DB_HOST=your_postgres_host
DB_PORT=5432
DB_NAME=your_database_name
DB_USER=your_database_user
DB_PASSWORD=your_database_password

# SSL (required for NeonDB and most cloud providers)
DB_SSL=true
```

> **Note:** The database connection is configured in `config/config.json` (generated from `config/config.example.json`). SSL is enabled by default for cloud deployments.

### Database Configuration Template

```json
{
  "development": {
    "username": "youruser",
    "password": "yourpassword",
    "database": "yourdatabase",
    "host": "yourhost",
    "port": 5432,
    "dialect": "postgres",
    "dialectOptions": {
      "ssl": {
        "require": true,
        "rejectUnauthorized": false
      }
    }
  }
}
```

### Running Migrations

```bash
# Apply all pending migrations
npx sequelize-cli db:migrate

# Undo the last migration
npx sequelize-cli db:migrate:undo

# Check migration status
npx sequelize-cli db:migrate:status
```

### `scripts/validarTSV.js` — TSV File Validator

Validates a TSV sales file **before** import to detect structural or data quality issues without writing to the database.

```bash
node scripts/validarTSV.js ./path/to/sales-file.txt
```

**What it checks:**

- File existence and size (warns if > 1 GB)
- All 38 required column headers are present
- Column count consistency per row
- Required fields are not empty (`Nro documento`, `Cliente factura`, `Item`, `Cantidad`, `Valor subtotal`)
- Date format validity (`DD/MM/YYYY`)
- Currency value format (`$X.XXX,XX`)

**Output example:**

```
✅ Archivo encontrado: ventastest.txt
ℹ️  Tamaño: 245.70 MB
ℹ️  Columnas detectadas: 38
✅ Todos los encabezados requeridos presentes
ℹ️  Validadas 1000 líneas...
...
═════════════════════════════════════════
VALIDACIÓN COMPLETADA
═════════════════════════════════════════

Estadísticas:
  📊 Total de líneas: 52840
  ✅ Líneas válidas: 52838
  ❌ Líneas con error: 2
  ⚠️  Campos vacíos: 0
  💰 Valores con moneda: 211360
```

**Exit codes:** `0` = file is ready for import · `1` = file has critical errors that must be corrected before importing.

When the script exits with code `1`, review the "Primeros errores detectados" section in the output. Each error includes the line number and a description (e.g., wrong column count, missing required field). Fix the source data in the TSV export and re-run the validator until it exits with code `0`.

## Performance

The SIGC DD API is engineered for high-volume commercial data workloads:

| Metric | Value |
|---|---|
| Maximum request payload | **6 GB** |
| Request / response timeout | **4 hours** |
| Import batch size | **10,000 rows** |
| Import transaction window | **5,000 rows** |
| Master table lookup | **O(1)** (in-memory Maps) |
| Password hashing | bcrypt with 10 salt rounds |

**Why these numbers matter:**

- A typical monthly sales export from the ERP can contain **50,000–500,000+ rows** and exceed 200 MB. The optimized importer handles this without exhausting memory by streaming line-by-line.
- Pre-loading all master data into `Map` structures at the start of each import eliminates repeated database round-trips, which would otherwise add milliseconds per row on large datasets.
- Batch `bulkCreate` operations with configurable transaction windows minimize the overhead of per-row commits while preserving recoverability.
